# Off-Chain Integration Guide

Midnight dApps split across two worlds: Compact contracts compile to ZK circuits that
execute on-chain, while TypeScript code orchestrates everything off-chain -- wallet
management, proof generation, state reads, and transaction submission. This guide covers
the off-chain half.

---

## 1. Architecture Overview

The compilation pipeline:

```
contract.compact
    |
    v  compact compile
managed/contract/
    ├── contract/index.cjs      # TypeScript bindings (Contract class)
    ├── keys/                   # Verifier keys (one per circuit)
    └── zkir/                   # Zero-knowledge intermediate representation
```

The compiled output produces a `Contract` class with typed methods for every exported
circuit. Calling a circuit method from TypeScript triggers this sequence:

1. **Witness execution** -- local TypeScript functions supply private inputs
2. **Circuit evaluation** -- the Compact runtime executes the circuit logic locally
3. **Proof generation** -- the proof server creates a ZK proof of correct execution
4. **Transaction assembly** -- public state deltas + proof are packaged into a transaction
5. **Signing** -- the wallet signs the transaction
6. **Submission** -- the transaction is submitted to the Midnight node
7. **Validation** -- the network verifies the proof and applies state changes

Off-chain components:

| Component | Role |
|-----------|------|
| TypeScript bindings | Type-safe circuit calls, generated by compiler |
| Witnesses | Local functions providing private inputs to circuits |
| Providers | Pluggable backends for state, proofs, wallet, network |
| Wallet SDK | Key management, signing, coin tracking |
| Proof server | Standalone process that generates ZK proofs |
| Indexer | GraphQL service that indexes on-chain state |

On-chain components:

| Component | Role |
|-----------|------|
| Ledger state | Public contract storage, updated by verified transactions |
| ZK proofs | Cryptographic proof that circuit executed correctly |
| Shielded pool | Private coin balances (Zswap UTXO model) |

## 2. Provider Pattern

Every Midnight dApp needs six providers. Together they form the runtime context that
connects your TypeScript code to the network. This is the standard initialization:

```typescript
import {
  levelPrivateStateProvider,
} from '@midnight-ntwrk/midnight-js-level-private-state-provider';
import {
  indexerPublicDataProvider,
} from '@midnight-ntwrk/midnight-js-indexer-public-data-provider';
import {
  httpClientProofProvider,
} from '@midnight-ntwrk/midnight-js-http-client-proof-provider';
import {
  NodeZkConfigProvider,
} from '@midnight-ntwrk/midnight-js-node-zk-config-provider';

const providers = {
  privateStateProvider: levelPrivateStateProvider<PrivateStates>({
    privateStateStoreName: config.privateStateStoreName,
  }),
  publicDataProvider: indexerPublicDataProvider(
    config.indexerUrl,
    config.indexerWsUrl,
  ),
  zkConfigProvider: new NodeZkConfigProvider<'increment' | 'decrement'>(
    config.zkConfigPath,
  ),
  proofProvider: httpClientProofProvider(config.proofServerUrl),
  walletProvider: walletAndMidnightProvider,
  midnightProvider: walletAndMidnightProvider,
};
```

### Provider Breakdown

**privateStateProvider** -- Manages per-user private state in a local database.

Each user has their own private state (secret keys, hidden balances, local data). This
provider persists it between sessions. Two implementations exist:

```typescript
// LevelDB (Node.js / Electron -- persistent storage)
import { levelPrivateStateProvider } from '@midnight-ntwrk/midnight-js-level-private-state-provider';

const psp = levelPrivateStateProvider<PrivateStates>({
  privateStateStoreName: 'my-app-private-state',
});

// In-memory (browser / testing -- no persistence)
// Roll your own Map-based implementation for browser dApps
const inMemoryStates = new Map<string, PrivateState>();
```

As of SDK 3.x, `levelPrivateStateProvider` requires encryption configuration. If you hit
errors about missing encryption, check the release notes for the exact API.

**publicDataProvider** -- Reads on-chain state via the indexer's GraphQL API.

The indexer watches the chain and exposes contract state through a GraphQL endpoint.
This provider subscribes to state changes so your dApp stays in sync.

```typescript
import { indexerPublicDataProvider } from '@midnight-ntwrk/midnight-js-indexer-public-data-provider';

const pdp = indexerPublicDataProvider(
  'http://127.0.0.1:8088/api/v3/graphql',   // HTTP endpoint
  'ws://127.0.0.1:8088/api/v3/graphql',      // WebSocket for subscriptions
);
```

**zkConfigProvider** -- Loads compiled circuit artifacts (verifier keys, zkir).

The proof server needs the circuit description and keys to generate proofs. This
provider loads them from disk (Node.js) or fetches them from a URL (browser).

```typescript
// Node.js -- reads from filesystem
import { NodeZkConfigProvider } from '@midnight-ntwrk/midnight-js-node-zk-config-provider';

const zkp = new NodeZkConfigProvider<'increment' | 'decrement'>(
  path.resolve(__dirname, '../contract/dist/managed/contract'),
);

// Browser -- fetches from URL
import { FetchZkConfigProvider } from '@midnight-ntwrk/midnight-js-fetch-zk-config-provider';

const zkp = new FetchZkConfigProvider<'increment' | 'decrement'>(
  window.location.origin,  // keys/ and zkir/ must be served from here
);
```

The type parameter is a union of your circuit names. The compiler generates these names
from your exported circuits.

**proofProvider** -- Sends circuit data to the proof server for ZK proof generation.

Proof generation is CPU-intensive (seconds to minutes depending on circuit complexity).
The proof server runs as a separate process, either locally or as a remote service.

```typescript
import { httpClientProofProvider } from '@midnight-ntwrk/midnight-js-http-client-proof-provider';

const pp = httpClientProofProvider('http://127.0.0.1:6300');
```

**walletProvider** -- Signs transactions and manages cryptographic keys.

**midnightProvider** -- Submits signed transactions to the Midnight node.

These two are typically the same object, created by the wallet SDK:

```typescript
const walletAndMidnightProvider = {
  coinPublicKey: wallet.coinPublicKey,
  balanceTx(tx: UnbalancedTransaction, newCoins: CoinInfo[]): Promise<BalancedTransaction> {
    return wallet.balanceTransaction(tx, newCoins);
  },
  submitTx(tx: ProvableTransaction): Promise<TransactionId> {
    return wallet.submitTransaction(tx);
  },
};
```

## 3. Witness Implementation

Witnesses are the bridge between private off-chain data and on-chain circuit execution.
They are declared in Compact (header only, no body) and implemented in TypeScript.

### Basic Structure

```typescript
import { WitnessContext } from '@midnight-ntwrk/compact-runtime';
import { Ledger } from '../managed/contract/contract/index.cjs';

// Your private state shape
interface PrivateState {
  readonly secretKey: Uint8Array;
  readonly localData: string;
}

// Witness implementations
export const witnesses = {
  // Simple: return a value from private state
  localSecretKey: ({
    privateState,
  }: WitnessContext<Ledger, PrivateState>): [PrivateState, Uint8Array] => {
    return [privateState, privateState.secretKey];
  },

  // With parameters: circuit passes arguments to witness
  division: (
    { privateState }: WitnessContext<Ledger, PrivateState>,
    x: bigint,
    y: bigint,
  ): [PrivateState, [bigint, bigint]] => {
    return [privateState, [x / y, x % y]];
  },

  // Mutating private state: return updated state
  recordAccess: ({
    privateState,
    ledger,
    contractAddress,
  }: WitnessContext<Ledger, PrivateState>): [PrivateState, Uint8Array] => {
    const updatedState = {
      ...privateState,
      lastAccessed: Date.now(),
    };
    return [updatedState, privateState.secretKey];
  },
};
```

### WitnessContext Fields

Every witness receives a `WitnessContext<Ledger, PrivateState>` with three fields:

| Field | Type | Description |
|-------|------|-------------|
| `privateState` | `PrivateState` | Current user's private state |
| `ledger` | `Ledger` | Snapshot of public ledger state at call time |
| `contractAddress` | `string` | Address of the deployed contract |

### Return Contract

Every witness MUST return a tuple: `[newPrivateState, returnValue]`.

- The first element is the (possibly updated) private state
- The second element is the value the circuit receives
- If you do not modify private state, return it unchanged

```typescript
// Correct: returns [PrivateState, Uint8Array]
localSecretKey: ({ privateState }) => [privateState, privateState.secretKey],

// Wrong: returning just the value
localSecretKey: ({ privateState }) => privateState.secretKey,  // WILL FAIL
```

### Type Mapping

Compact types map to TypeScript types in witnesses:

| Compact type | TypeScript type |
|-------------|----------------|
| `Field` | `bigint` |
| `Boolean` | `boolean` |
| `Bytes<N>` | `Uint8Array` |
| `Uint<N>` | `bigint` |
| `Opaque<"string">` | `string` |
| `Maybe<T>` | `{ is_some: boolean; value: T }` |
| `Enum` | `bigint` (variant index) |
| `Struct` | Object with matching field names |

### Security Considerations

Witness code is UNVERIFIED. It runs locally on the user's machine, outside of the ZK
proof. This means:

1. **Witnesses can lie.** A malicious user can modify witness code to return anything.
   The circuit must validate witness outputs using `disclose()` and `assert()`.
2. **Witnesses can access anything.** Browser APIs, network requests, databases, file
   system -- anything available in JavaScript. This is a feature: use it for off-chain
   data fetching, user prompts, or external service calls.
3. **Witnesses are private.** The network never sees witness inputs or code. Only the
   resulting ZK proof is published.

```typescript
// Example: witness that fetches a price from an API
getPrice: async ({ privateState }) => {
  const resp = await fetch('https://api.example.com/price');
  const { price } = await resp.json();
  return [privateState, BigInt(Math.floor(price * 1e8))];
},
```

The circuit MUST validate the result (e.g., check it falls within an expected range
using `assert(disclose(price < maxPrice), "Price out of range")`).

## 4. Wallet SDK Integration

The Midnight wallet SDK is modular. Each package handles a specific concern:

| Package | Purpose |
|---------|---------|
| `@midnight-ntwrk/wallet-sdk-facade` | Orchestrates all wallet components |
| `@midnight-ntwrk/wallet-sdk-hd` | HD key derivation (BIP32-based) |
| `@midnight-ntwrk/wallet-sdk-shielded` | Shielded (private) coin operations |
| `@midnight-ntwrk/wallet-sdk-unshielded-wallet` | Unshielded (transparent) coin operations |
| `@midnight-ntwrk/wallet-sdk-dust-wallet` | Fee payment operations |
| `@midnight-ntwrk/wallet-sdk-address-format` | Bech32m address encoding/decoding |

### Building a Wallet

```typescript
import { WalletBuilder } from '@midnight-ntwrk/wallet-sdk-facade';
import { NetworkId } from '@midnight-ntwrk/midnight-js-network-id';

// IMPORTANT: WalletBuilder.buildFromSeed expects a hex-encoded BIP32 seed,
// NOT a mnemonic phrase. Convert mnemonic to seed first.
const seed = '0a1b2c3d...'; // 64-byte hex string (128 chars)

const wallet = await WalletBuilder.buildFromSeed(
  seed,
  NetworkId.TestNet,     // or NetworkId.Undeployed for local
  'http://127.0.0.1:8088/api/v3/graphql',   // indexer HTTP
  'ws://127.0.0.1:8088/api/v3/graphql',     // indexer WS
  'http://127.0.0.1:6300',                   // proof server
  'ws://127.0.0.1:9944',                     // node RPC
);
```

### Waiting for Wallet Sync

After creation, the wallet must sync with the chain to discover existing coins and
state. This is essential before any transaction:

```typescript
import * as Rx from 'rxjs';
import { nativeToken } from '@midnight-ntwrk/ledger-v7';

const waitForFunds = (wallet: Wallet): Promise<Map<string, bigint>> =>
  Rx.firstValueFrom(
    wallet.state().pipe(
      // Wait until sync is complete
      Rx.filter((state) => state.syncProgress?.synced === true),
      // Extract balances
      Rx.map((s) => s.balances),
      // Wait until we have a positive native token balance
      Rx.filter((balances) => balances != null && balances[nativeToken()] > 0n),
    ),
  );

console.log('Waiting for wallet to sync and find funds...');
const balances = await waitForFunds(wallet);
console.log(`Balance: ${balances[nativeToken()]} tDUST`);
```

### Getting the Wallet Address

```typescript
const address = wallet.address; // bech32m-encoded address string
const coinPublicKey = wallet.coinPublicKey; // raw public key for coin operations
```

### Creating the Combined Provider

Most dApps need both wallet and midnight provider functionality. Build a combined object:

```typescript
import { UnbalancedTransaction, BalancedTransaction } from '@midnight-ntwrk/midnight-js-types';
import { CoinInfo } from '@midnight-ntwrk/ledger-v7';

const walletAndMidnightProvider = {
  coinPublicKey: wallet.coinPublicKey,

  async balanceTx(
    tx: UnbalancedTransaction,
    newCoins: CoinInfo[],
  ): Promise<BalancedTransaction> {
    return wallet.balanceTransaction(tx, newCoins);
  },

  async submitTx(tx: ProvableTransaction): Promise<TransactionId> {
    return wallet.submitTransaction(tx);
  },
};
```

## 5. Transaction Flow

### Circuit Call (Standard Path)

The most common operation is calling an exported circuit:

```typescript
import { Contract, deployContract, findDeployedContract } from '@midnight-ntwrk/midnight-js-contracts';
import { ContractType } from '../managed/contract/contract/index.cjs';

// Deploy a new contract
const deployedContract = await deployContract(providers, {
  contract: ContractType,
  initialPrivateState: { secretKey: mySecretKey },
  witnesses,
});

console.log(`Deployed at: ${deployedContract.deployTxData.contractAddress}`);

// Or find an existing deployment
const joinedContract = await findDeployedContract(providers, {
  contractAddress: knownAddress,
  contract: ContractType,
  initialPrivateState: { secretKey: mySecretKey },
  witnesses,
});

// Call a circuit -- this triggers the full proof/sign/submit pipeline
const txResult = await deployedContract.callTx.increment();
console.log(`Transaction ID: ${txResult.txId}`);
```

### What Happens During callTx

When you call `deployedContract.callTx.increment()`:

1. The Compact runtime invokes witnesses to collect private inputs
2. The circuit executes locally, producing public state changes
3. The unbalanced transaction is sent to the wallet for balancing (adding fee coins)
4. The balanced transaction is sent to the proof server for ZK proof generation
5. The wallet signs the proven transaction
6. The signed transaction is submitted to the Midnight node
7. The promise resolves when the node accepts the transaction

### Programmatic Token Transfer

For direct tNight transfers (v7 pattern — requires `signRecipe`, see gotcha #59):

```typescript
import { unshieldedToken } from '@midnight-ntwrk/ledger-v7';

const recipe = await wallet.transferTransaction(
  [{ type: "unshielded", outputs: [{
    amount: 100_000_000n,  // 100 tNight
    receiverAddress: 'mn_addr_preprod1...',
    type: unshieldedToken().raw,
  }] }],
  { shieldedSecretKeys, dustSecretKey },
  { ttl: new Date(Date.now() + 30 * 60 * 1000), payFees: true },
);
// REQUIRED: signRecipe before finalizing (gotcha #59 — error 139 without this)
const signed = await wallet.signRecipe(recipe, (payload) => keystore.signData(payload));
const finalized = await wallet.finalizeRecipe(signed);
const txId = await wallet.submitTransaction(finalized);
```

Multi-output transfers work — include multiple entries in the `outputs` array (gotcha #62).

**v8 migration:** In WalletFacade 3.0.0-rc.0, wallet initialization uses `WalletFacade.init()` instead of the constructor. Import from `@midnight-ntwrk/ledger-v8` instead of `ledger-v7`. See gotcha #74 for full migration details.

### Reading Contract State

You do not need a transaction to read public state:

```typescript
// Read current ledger state from the indexer
const currentState = await deployedContract.callQry.getBalance(myAddress);
```

Query calls (`callQry`) execute locally against the indexer's state snapshot. They do
not produce transactions or ZK proofs.

## 6. Vite Configuration for Browser dApps

Midnight SDK packages use WebAssembly and top-level await. Standard Vite configuration
will not work. You need three plugins:

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import wasm from 'vite-plugin-wasm';
import topLevelAwait from 'vite-plugin-top-level-await';
import { viteCommonjs } from '@originjs/vite-plugin-commonjs';

export default defineConfig({
  build: {
    target: 'esnext',  // Required for top-level await
    minify: false,      // Minification can break WASM imports
  },

  plugins: [
    wasm(),              // Handle .wasm imports
    react(),             // Or your framework of choice
    viteCommonjs(),      // Some SDK packages are CJS-only
    topLevelAwait(),     // Required by SDK initialization code
  ],

  optimizeDeps: {
    esbuildOptions: {
      target: 'esnext',  // Match build target
    },
    // You may need to exclude certain SDK packages from pre-bundling
    exclude: ['@midnight-ntwrk/compact-runtime'],
  },

  // Some SDK packages use Node.js APIs that need polyfills
  resolve: {
    alias: {
      // Add polyfills as needed
    },
  },
});
```

Install the required dev dependencies:

```bash
npm install -D vite-plugin-wasm vite-plugin-top-level-await @originjs/vite-plugin-commonjs
```

### Browser vs Node.js Differences

| Concern | Node.js | Browser |
|---------|---------|---------|
| Private state | `levelPrivateStateProvider` (LevelDB) | Custom in-memory or IndexedDB |
| ZK config | `NodeZkConfigProvider` (filesystem) | `FetchZkConfigProvider` (HTTP) |
| Proof server | Local process on `localhost:6300` | Remote service or same-origin proxy |
| Wallet | Direct SDK instantiation | DApp Connector (browser extension) |

### DApp Connector (Browser Extension Wallet)

For browser dApps, users connect their wallet via the Midnight DApp Connector:

```typescript
// Check if the Midnight wallet extension is available
declare global {
  interface Window {
    midnight?: {
      mnLace?: {
        isEnabled: () => Promise<boolean>;
        enable: () => Promise<DAppConnectorAPI>;
      };
    };
  }
}

// Connect to wallet
const api = await window.midnight!.mnLace!.enable();
const walletState = await api.state();
```

Note: The DApp Connector API does NOT expose a transfer method. To transfer tokens from
a browser dApp, you must deploy a contract with a transfer circuit. This is a known
limitation of the current API.

## 8. Network Configuration

### Environments

| Network | Node RPC | Indexer | Use Case |
|---------|----------|---------|----------|
| Standalone (local) | `ws://127.0.0.1:9944` | `http://127.0.0.1:8088/api/v3/graphql` | Development |
| Preview | Public endpoints (see docs) | Public endpoints | Early testing (unstable) |
| Preprod | Public endpoints (see docs) | Public endpoints | Pre-production testing |

### Local Development Setup

The easiest way to run a local network is with Brick Towers' Docker Compose:

```bash
git clone https://github.com/bricktowers/midnight-local-network.git
cd midnight-local-network
docker compose up -d
```

This starts:
- Midnight node on port 9944 (WebSocket RPC)
- Indexer on port 8088 (GraphQL HTTP + WS)
- Proof server on port 6300 (HTTP)

### Indexer GraphQL

The indexer exposes contract state via GraphQL:

```graphql
# Query contract state
query {
  contractState(contractAddress: "0x...") {
    state
    blockHeight
  }
}

# Subscribe to state changes
subscription {
  contractStateChanged(contractAddress: "0x...") {
    state
    blockHeight
  }
}
```

The `publicDataProvider` wraps these queries automatically. You rarely need to write
GraphQL directly.

### Proof Server

The proof server is a standalone process that generates ZK proofs. It needs access to
the proving parameters (large files, hundreds of MB).

```bash
# Brick Towers pre-baked proof server (recommended -- avoids parameter download)
docker pull bricktowers/midnight-proof-server:latest
docker run -p 6300:6300 bricktowers/midnight-proof-server:latest
```

Proof generation time depends on circuit complexity (`k` value). A simple counter
circuit takes 2-5 seconds. Complex circuits with large Maps or many operations can take
30+ seconds.

## 9. Deployment

### Compilation

```bash
# Compile Compact contract to circuits + TypeScript bindings
compact compile src/contract.compact src/managed/contract

# Output structure:
# src/managed/contract/
#   ├── contract/
#   │   └── index.cjs          # TypeScript bindings
#   ├── keys/
#   │   ├── increment.verifier  # Verifier key per circuit
#   │   └── decrement.verifier
#   └── zkir/
#       ├── increment.zkir      # Circuit description per circuit
#       └── decrement.zkir
```

### Contract Deployment (Programmatic)

```typescript
import { deployContract } from '@midnight-ntwrk/midnight-js-contracts';
import { Contract as ContractType } from '../managed/contract/contract/index.cjs';

const deployed = await deployContract(providers, {
  contract: ContractType,
  initialPrivateState: {
    secretKey: new Uint8Array(32), // your initial private state
  },
  witnesses,
});

const contractAddress = deployed.deployTxData.contractAddress;
console.log(`Contract deployed at: ${contractAddress}`);
```

### Joining an Existing Contract

```typescript
import { findDeployedContract } from '@midnight-ntwrk/midnight-js-contracts';

const contract = await findDeployedContract(providers, {
  contractAddress: '0x...',
  contract: ContractType,
  initialPrivateState: myPrivateState,
  witnesses,
});

// Now call circuits
const result = await contract.callTx.increment();
```

### Browser Deployment

For browser dApps, the verifier keys and zkir files must be served as static assets:

```bash
# Copy artifacts to your web build output
cp -r src/managed/contract/keys ./dist/keys
cp -r src/managed/contract/zkir ./dist/zkir
```

Then configure `FetchZkConfigProvider` to load them:

```typescript
import { FetchZkConfigProvider } from '@midnight-ntwrk/midnight-js-fetch-zk-config-provider';

const zkConfigProvider = new FetchZkConfigProvider<'increment' | 'decrement'>(
  window.location.origin, // serves /keys/* and /zkir/* from dist/
);
```

## 10. Version Compatibility

Midnight's ecosystem has strict version coupling. Mismatched versions produce cryptic
errors (proof failures, serialization errors, type mismatches).

### The Compatibility Rule

The node, indexer, proof server, and ALL SDK npm packages must be from the same release.
Mixing versions will fail silently or produce invalid proofs.

### How to Stay Aligned

1. **Start from an official example.** Clone `example-counter` or `example-bboard` and
   use its exact `package.json` versions.
2. **Check the compatibility matrix.** The official matrix lives at
   `https://docs.midnight.network/relnotes/comp-matrix` -- but it is often outdated.
3. **Lock versions.** Use exact versions in `package.json`, not ranges:

```json
{
  "dependencies": {
    "@midnight-ntwrk/midnight-js-contracts": "3.0.0",
    "@midnight-ntwrk/compact-runtime": "0.14.0",
    "@midnight-ntwrk/wallet-sdk-facade": "1.0.0"
  }
}
```

4. **Use Brick Towers' local network.** Their Docker images are tested against specific
   SDK versions. Check their README for the compatible SDK version.

### Version Discovery

When versions are unclear, check what the official examples use:

```bash
# Clone the latest example and inspect its dependencies
git clone https://github.com/midnightntwrk/example-counter.git
cat example-counter/counter-cli/package.json | grep midnight
```

## 11. Common Issues

### signRecipe Bug (wallet-sdk-unshielded-wallet 1.0.0)

The `signRecipe` method in wallet-sdk-unshielded-wallet 1.0.0 has a known bug that
requires manual intent signing. Workaround:

```typescript
// If wallet.balanceTransaction fails with a signing error,
// you may need to manually sign the intent:
const balanced = await wallet.balanceTransaction(unbalancedTx, newCoins);
// If this throws, check if you need to downgrade or upgrade the wallet SDK
```

Monitor the Midnight Discord `#sdk-support` channel for patches.

### nativeToken() vs unshieldedToken().raw Length Mismatch

`nativeToken()` returns a 68-character hex string, while `unshieldedToken().raw` returns
64 characters. These are NOT interchangeable:

```typescript
import { nativeToken } from '@midnight-ntwrk/ledger-v7';

// Correct: use nativeToken() for balance lookups
const balance = balances[nativeToken()];

// Wrong: comparing nativeToken() output with unshieldedToken().raw
// They have different lengths and will never match
```

### DApp Connector Transfer Limitation

The browser DApp Connector API does NOT expose a `transfer` method. You cannot
programmatically transfer tokens from a user's wallet through the connector.

Workaround: deploy a contract with a circuit that performs the transfer:

```compact

  // Build transfer logic into your contract
  sendImmediate(nativeCoin(), left<ZswapCoinPublicKey, ContractAddress>(recipient), amount);
}
```

If you see "Failed to fetch" or "CompileError" for WASM modules:

### Watching for State Changes

Subscribe to contract actions via the indexer WebSocket:

```typescript
import { WebSocket } from "ws";

function watchContract(address: string, wsUrl: string) {
  const ws = new WebSocket(wsUrl, "graphql-ws");

  ws.on("open", () => {
    ws.send(JSON.stringify({ type: "connection_init" }));
    ws.send(JSON.stringify({
      id: "1",
      type: "start",
      payload: {
        query: `subscription {
          contractStateChanged(contractAddress: "${address}") {
            state
            blockHeight
          }
        }`,
      },
    }));
  });

    if (msg.type === "data") {
      console.log("State changed:", msg.payload.data);
    }
  });

  return ws;
}
```

## 13. Error Handling & Recovery

Off-chain code must handle failures gracefully. The proof-generate-sign-submit pipeline
has multiple failure points.

### Transaction Lifecycle Errors

```typescript
async function safeCircuitCall(
  contract: DeployedContract,
  circuitName: string,
  ...args: any[]
) {
  try {
    const result = await contract.callTx[circuitName](...args);
    return { success: true, txId: result.txId };
  } catch (error: any) {
    const msg = error.message || String(error);

    if (msg.includes("proof") || msg.includes("Proof")) {
      // Proof generation failed — circuit may be too complex or proof server down
      return { success: false, stage: "proof", error: msg, retryable: true };
    }

    if (msg.includes("balance") || msg.includes("DUST") || msg.includes("insufficient")) {
      // Insufficient DUST for fees — wait for DUST to accrue or top up tNight
      return { success: false, stage: "balance", error: msg, retryable: true };
    }

    if (msg.includes("timeout") || msg.includes("ETIMEDOUT")) {
      // Network timeout — node or indexer may be down
      return { success: false, stage: "network", error: msg, retryable: true };
    }

    if (msg.includes("assert") || msg.includes("Circuit")) {
      // Circuit assertion failed — this is a logic error, not retryable
      return { success: false, stage: "circuit", error: msg, retryable: false };
    }

    return { success: false, stage: "unknown", error: msg, retryable: false };
  }
}
```

### Wallet State Desync Recovery

When the wallet falls out of sync with the chain:

```typescript
async function resyncWallet(wallet: WalletFacade): Promise<void> {
  // Wait for wallet to catch up to chain head
  const state = await Rx.firstValueFrom(
    wallet.state().pipe(
      Rx.filter((s: any) => s.isSynced),
      Rx.timeout(300_000), // 5 minutes max
    ),
  );

  if (!state.isSynced) {
    throw new Error("Wallet failed to sync — check indexer connectivity");
  }
}
```

### Proof Server Health Check

```typescript
async function checkProofServer(url: string): Promise<boolean> {
  try {
    const resp = await fetch(`${url}/health`, { signal: AbortSignal.timeout(5000) });
    return resp.ok;
  } catch {
```

### Common Failure Patterns

| Symptom | Likely Cause | Recovery |
|---------|-------------|----------|
| "context.ctor is not a constructor" | `compact-js` version mismatch between packages | Pin `compact-js` to match `midnight-js-contracts` dependency |
| `LEVEL_LOCKED` | Two processes accessing same LevelDB directory | Use separate directories or in-memory provider |
| Proof server timeout | Circuit k-value too high or proof server overloaded | Increase timeout, optimize circuit, or restart proof server |
| "insufficient DUST" | DUST balance too low for transaction fee | Wait for DUST to accrue from tNight, or request more tNight from faucet |
| Wallet sync stuck | Indexer behind chain head or WebSocket disconnected | Check indexer health, reconnect WebSocket |
| Transaction stuck | Block production paused or tx rejected by mempool | Check node peer count, verify tx was accepted |

## Full Working Example: Counter dApp (Node.js CLI)

Putting it all together -- a complete Node.js CLI application that deploys and interacts
with a counter contract:

```typescript
import * as path from 'path';
import * as Rx from 'rxjs';
import { deployContract } from '@midnight-ntwrk/midnight-js-contracts';
import { levelPrivateStateProvider } from '@midnight-ntwrk/midnight-js-level-private-state-provider';
import { indexerPublicDataProvider } from '@midnight-ntwrk/midnight-js-indexer-public-data-provider';
import { httpClientProofProvider } from '@midnight-ntwrk/midnight-js-http-client-proof-provider';
import { NodeZkConfigProvider } from '@midnight-ntwrk/midnight-js-node-zk-config-provider';
import { WalletBuilder } from '@midnight-ntwrk/wallet-sdk-facade';
import { NetworkId } from '@midnight-ntwrk/midnight-js-network-id';
import { nativeToken } from '@midnight-ntwrk/ledger-v7';
import { Contract as CounterContract } from '../managed/counter/contract/index.cjs';

// Configuration for local standalone network
const config = {
  indexer: 'http://127.0.0.1:8088/api/v3/graphql',
  indexerWs: 'ws://127.0.0.1:8088/api/v3/graphql',
  node: 'ws://127.0.0.1:9944',
  proofServer: 'http://127.0.0.1:6300',
  zkConfigPath: path.resolve(__dirname, '../managed/counter'),
};

// Private state type
interface CounterPrivateState {
  readonly secretKey: Uint8Array;
}

// Witnesses
const witnesses = {
  localSecretKey: ({
    privateState,
  }: { privateState: CounterPrivateState }): [CounterPrivateState, Uint8Array] => {
    return [privateState, privateState.secretKey];
  },
};

async function main() {
  // 1. Build wallet from seed
  const seed = process.env.MIDNIGHT_SEED!;
  console.log('Building wallet...');
  const wallet = await WalletBuilder.buildFromSeed(
    seed,
    NetworkId.Undeployed,
    config.indexer,
    config.indexerWs,
    config.proofServer,
    config.node,
  );

  // 2. Wait for wallet sync
  console.log('Waiting for wallet sync...');
  await Rx.firstValueFrom(
    wallet.state().pipe(
      Rx.filter((s) => s.syncProgress?.synced === true),
      Rx.filter((s) => s.balances != null && s.balances[nativeToken()] > 0n),
    ),
  );
  console.log('Wallet synced.');

  // 3. Build providers
  const walletAndMidnightProvider = {
    coinPublicKey: wallet.coinPublicKey,
    balanceTx: (tx: any, coins: any) => wallet.balanceTransaction(tx, coins),
    submitTx: (tx: any) => wallet.submitTransaction(tx),
  };

  const providers = {
    privateStateProvider: levelPrivateStateProvider<{ counterPrivateState: CounterPrivateState }>({
      privateStateStoreName: 'counter-private-state',
    }),
    publicDataProvider: indexerPublicDataProvider(config.indexer, config.indexerWs),
    zkConfigProvider: new NodeZkConfigProvider<'increment' | 'decrement'>(config.zkConfigPath),
    proofProvider: httpClientProofProvider(config.proofServer),
    walletProvider: walletAndMidnightProvider,
    midnightProvider: walletAndMidnightProvider,
  };

  // 4. Deploy contract
  console.log('Deploying counter contract...');
  const deployed = await deployContract(providers, {
    contract: CounterContract,
    initialPrivateState: {
      secretKey: new Uint8Array(32),
    },
    witnesses,
  });
  console.log(`Deployed at: ${deployed.deployTxData.contractAddress}`);

  // 5. Call increment circuit
  console.log('Calling increment...');
  const result = await deployed.callTx.increment();
  console.log(`Transaction: ${result.txId}`);

  // 6. Read current state
  const state = deployed.state;
  console.log(`Counter value: ${state.counter}`);
}

main().catch(console.error);
```

This example assumes a running local network (node + indexer + proof server) and a
funded wallet seed in the `MIDNIGHT_SEED` environment variable. See section 8 for
local network setup.
